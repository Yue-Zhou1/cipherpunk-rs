# V3 Hybrid Audit Workstation Design

> **Status:** Approved design basis for `plan-v3`
> **Date:** March 12, 2026
> **Audience:** Product, platform, UI, and engine engineers

## Context

`implementation-plan-v2.md` defines a guided audit wizard that collects source/configuration, runs an audit pipeline, and renders results. That shape is useful for intake and batch execution, but it is not the right primary interface for the target users:

- security engineers reviewing cryptography, ZK, P2P, and consensus-heavy codebases
- auditors who move repeatedly between source code, architecture relationships, helper tools, traces, notes, and findings
- engineers who need reproducible evidence, not only summarized output

`v3` therefore shifts the product from a pure "audit workflow app" into a hybrid product:

- a `Wizard` for intake and session creation
- a `Workstation` for interactive security investigation

## Goals

- Keep the existing intake strengths from `v2`: source pinning, scope selection, optional inputs, confirmation, and exportable configs.
- Add a VSCode-like security workstation as the main analyst environment after session creation.
- Let AI act as a structured analyst copilot: map code to graphs, produce an initial security overview, choose relevant checklists, and prepare tool runs.
- Keep trust boundaries clear: AI can propose, summarize, prioritize, and scaffold; deterministic tools and human reviewers decide what is verified.
- Support Rust, Circom, Cairo, and distributed protocol projects through a shared project representation and language/domain-specific playbooks.
- Make verified outputs reproducible, persistent, searchable, and suitable for commercial-grade security reporting.

## Non-Goals For Initial V3

- Replacing human auditors with autonomous bug adjudication
- Full automatic theorem-prover authoring for Lean as a launch blocker
- Perfect dynamic value tracing of sensitive secrets by default
- Turning every AI note into an exported security finding

## Product Shape

### Wizard Mode

Wizard mode remains the front door for:

- source resolution and pinning
- audit scope and build variant selection
- optional inputs such as specs, invariants, and previous audits
- workspace confirmation
- initial session creation
- batch audit launch

Wizard mode no longer represents the entire product. It creates an `Audit Session` and then hands the engineer into the workstation.

### Workstation Mode

Workstation mode becomes the primary analyst interface:

- left panel: project explorer, session outline, checklist domains, saved notes/candidates/findings
- center panel: code editor plus graph lenses
- right panel: helper toolbench and AI panels
- bottom panel: execution console, logs, traces, evidence previews, replay output

The workstation is persistent. The engineer can leave and return to a session without losing context.

## AI Responsibility Model

`v3` uses `AI as structured analyst copilot`.

### AI Is Allowed To:

- build code-to-graph projections from parser and semantic outputs
- generate an initial architecture and security overview
- classify which audit domains and checklists apply
- prioritize hotspots and propose hypotheses
- draft review notes and unverified candidates
- prepare harnesses, configs, scenario templates, and repro scaffolding
- summarize traces, solver outputs, and long execution logs
- draft remediation ideas and regression test outlines

### AI Is Not Allowed To:

- silently decide that a bug is real
- silently assign final severity for exported verified findings
- upgrade its own reasoning into `Verified`
- remove uncertainty labels from unverified material

### Record Classes

The system distinguishes three record classes:

- `Verified Finding`
  Backed by deterministic tool output or explicit human confirmation.
- `Unverified Candidate`
  Generated by AI or heuristics and waiting for validation.
- `Review Note`
  Architecture/security observations, checklist commentary, or hotspot notes without a bug claim.

## Shared Backend Architecture

Both UI modes depend on the same shared backend objects.

### Core Objects

- `Project Snapshot`
  Pinned repo/archive/local source state with commit hash or content hash, build variants, detected frameworks, and scope.
- `Project IR`
  Canonical representation of files, symbols, features, framework nodes, and redacted dataflow relationships.
- `Audit Session`
  Persistent workspace bound to one project snapshot. Stores selected domains, job history, notes, candidates, findings, evidence references, and UI state.
- `Analysis Job`
  Concrete executable unit for graph extraction, overview generation, checklist execution, tool actions, exports, and replay.
- `Evidence Artifact`
  Harnesses, traces, solver inputs/outputs, fuzz corpora, scripts, logs, and provenance.
- `Knowledge Base`
  Human-curated playbooks, tool-routing rules, domain checklists, prior adjudicated cases, and policy rules.

### Architectural Rule

Every tool run and every AI-generated note attaches back to the same `Audit Session` and `Project IR`. That turns the system into a coherent workstation instead of a collection of disconnected features.

## Project IR And Code-To-Graph Mapping

The workstation needs a shared representation that is richer than a file tree.

### Required Graph Views

- `File graph`
  Dependencies and references between files/modules/crates.
- `Feature graph`
  Relationships between features, entry points, modules, and functions.
- `Dataflow graph`
  Redacted view of inputs, intermediate states, transformations, and sinks.
- `Framework graph`
  Halo2/Circom/Cairo/distributed-specific nodes and edges for domain-aware review.

### Recommended Technical Approach

- `tree-sitter` for multi-language parsing and syntactic indexing
- `rust-analyzer`-derived semantic information for Rust navigation and symbol relationships
- language adapters for Rust, Circom, and Cairo behind a shared graph-builder trait
- `Cytoscape.js` in the workstation for interactive graph rendering

### Safety Rule

Graphing defaults to structure and type information. Sensitive concrete values remain redacted unless the engineer explicitly opts in under policy controls.

## Tooling And Knowledge System

Helper tools become first-class capabilities rather than ad hoc buttons.

### Tool System

- project and framework detection selects candidate tool families
- domain playbooks define which checklist packs apply
- AI selects and orders relevant domains
- deterministic engines and sandboxed jobs execute concrete checks
- each run writes artifacts and structured results into the session

### Initial Tooling Scope

- Rust: Kani, Z3, cargo-fuzz/libFuzzer, Miri
- Circom: graph extraction, constraint checks, solver-backed under-constraint checks
- Cairo/Starknet: graphing, checklist routing, external plugin slot for ecosystem tools
- Distributed/P2P/consensus: MadSim/chaos-style scenarios, invariant monitoring, replay artifacts
- Lean: experimental external tool adapter, not a launch blocker for `v3`

### Knowledge Base Scope

- `Playbooks`
  Project-shape guides such as Rust crypto library, Circom circuit repo, Cairo/Starknet contract, libp2p consensus node.
- `Checklist packs`
  Crypto, ZK, serialization, witness handling, proving/verifying boundaries, P2P/networking, consensus safety/liveness, economics, supply chain.
- `Adjudicated cases`
  Prior true positives, false positives, repro patterns, and useful tool sequences.
- `Policy rules`
  AI permissions, escalation rules, and verification requirements.

## Persistence, Memory, And Search

The system needs persistent local-first memory, not only temporary run output.

### Storage Model

- relational store for sessions, jobs, notes, candidates, findings, policies, and adjudication history
- filesystem/object storage for artifacts and evidence packs
- full-text search index for read-heavy querying
- optional vector retrieval for semantic guidance, never as the source of truth

### Recommended Default

- SQLite + FTS5 for local desktop persistence and fast query
- filesystem storage under a managed `.audit-sessions/` root for large artifacts

## Production Requirements

### Required Properties

- `Reproducibility`
  Verified results are replayable from pinned source, pinned tools, pinned containers, and saved artifacts.
- `Isolation`
  Risky runs execute in sandboxed local or remote workers with resource and network controls.
- `Provenance`
  Artifacts record tool versions, container digests, timestamps, seeds, config, and source snapshot.
- `Persistence`
  Sessions, graphs, notes, candidates, findings, and adjudication history survive restarts.
- `Redaction`
  Sensitive values are protected in traces, logs, previews, and AI prompts by default.
- `Observability`
  Job state, worker health, failures, retries, and audit logs are inspectable.
- `Human control`
  Engineers can confirm, reject, suppress, annotate, and export candidates/findings.
- `Scalable compute`
  Expensive formal and distributed jobs can run on remote workers.

## Delivery Strategy

Deliver `v3` in this order:

1. new core schemas and persistent session model
2. project IR and graph extraction
3. AI overview and checklist planning
4. wizard-to-session handoff and workstation shell
5. targeted toolbench actions and evidence wiring
6. review queue, reporting, and feedback memory
7. remote workers and production hardening

This ordering preserves `v2` usability while allowing `v3` capabilities to land incrementally.

## Summary

`v3` is not a cosmetic extension of `v2`. It is a hybrid redesign that keeps the wizard for intake, adds a persistent workstation for real analyst work, expands AI into a structured copilot role, and treats graphing, knowledge, evidence, and reproducible tooling as first-class systems.
